# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Telegram bot that exposes a real Claude Code agent over chat. A user messages the bot; the bot drives the Claude Agent SDK on the host machine, streams the reply back live, and gates risky tool calls behind inline approval buttons. **The bot can read/write/run anything on the host**, the only access control is the `ALLOWED_USER_IDS` allow-list enforced in `src/auth.ts`.

## Commands

```bash
npm run dev        # concurrently: tsx watch (bot) + vite build --watch (panel), both stay fresh
npm run build      # tsc -> dist/
npm start          # node dist/index.js (requires build first)
npm run typecheck  # tsc --noEmit
```

There is no test suite, linter config, or single-test runner. `typecheck` (strict mode, with `noUnusedLocals`/`noUnusedParameters`) is the only automated check, run it after changes.

Requires Node >= 20. Config comes from `.env` (copy from `.env.example`); the process exits at startup with a printed list of issues if required vars are missing (see `parseConfig` in `src/config.ts`).

### Deployment scripts (`scripts/`)

`cct-install.sh` is the self-contained `curl | bash` wizard (installs prerequisites, clones, builds, configures `.env`, optionally installs the service); it does **not** assume a checkout and reads prompts from `/dev/tty` so it stays interactive when piped. The rest operate on an existing checkout: `run.sh` (launcher, also the service `ExecStart`), `update.sh` (pull + build + restart-if-service), `install-service.sh` / `uninstall-service.sh` / `agentctl.sh` (all OS-dispatchers to `linux/` = systemd, `macos/` = launchd). Keep the dispatchers thin and put platform specifics in the `linux/`/`macos/` impls. The hosted `https://gyorgy.sh/cct-install.sh` is a rewrite of the raw GitHub file, **bump both together** when changing the installer.

## Architecture

ESM throughout (`"type": "module"`), so **relative imports must use the `.js` extension** even though sources are `.ts`.

Every turn opens with a "💭 Working on it…" ack message (`handleUserPrompt`): in `edit` mode it *is* the streamed reply (edited in place); in the `rich`/`draft` modes it's a standalone placeholder deleted in the `finally`. Streamed text is run through `normalizeAgentText` (`formatting.ts`) first, Claude Code's TUI pads with U+00A0/U+202F/U+2007 instead of ASCII spaces, which render badly in Telegram, so they're folded back to a normal space.

Request lifecycle for one user message (`handleUserPrompt` in `src/bot.ts`):
1. `auth.ts` middleware drops anyone not in `allowedUserIds`.
2. A per-chat `Session` (`src/session/manager.ts`) holds `sessionId` (Claude resume token), `cwd`, `busy` flag, `abort` controller, `mode`, the per-session "always allow" tool set, and `usage` counters. The durable fields (everything but `busy`/`abort`) are persisted to `STATE_FILE` (JSON, default `data/state.json`) via `src/session/store.ts`, loaded on boot, written debounced on mutation, and flushed on shutdown, so sessions survive a restart.
3. A placeholder message is sent, wrapped in a `TelegramStreamer`.
4. `runTurn` (`src/claude/runner.ts`) calls the SDK `query()` and iterates its async message stream, fanning events to callbacks: `onText` (streaming deltas), `onToolUse` (status line), `onSessionId` (capture resume token), and the final `result` (cost/duration).
5. Streamer edits the message in place (throttled) and the loop ends; `busy`/`abort` are cleared in `finally`.

Key cross-cutting pieces:
- **Permission flow** (`src/telegram/permissions.ts` + `canUseTool` in `bot.ts`): read-only tools in `AUTO_ALLOWED_TOOLS` (`runner.ts`) run automatically; everything else posts Approve/Deny/Always-allow buttons and the `canUseTool` promise blocks until a `callback_query` resolves it (or it times out → deny). "Always" adds the tool to `session.sessionAllowedTools`. Pending requests are keyed by a random id embedded in the callback data (`appr:<id>:<action>`).
- **Modes**: `safe` maps to SDK `permissionMode: "default"` (interactive approval); `auto` maps to `"bypassPermissions"` (no prompts). Set via `/mode`.
- **Streaming**, three backends behind `STREAM_MODE` (config), all implementing the `Streamer` interface (`appendText`/`setStatus`/`finalize`), selected in `handleUserPrompt`:
  - `rich` (default): Bot API 10.1 Rich Messages. Streams `sendRichMessageDraft` (Claude's markdown → `InputRichMessage.markdown`, Telegram parses the structure) and finalizes with `sendRichMessage`. `richDraftStreamer.ts`.
  - `draft`: Bot API 9.3 `sendMessageDraft`, plain-text animated preview, finalized as a normal formatted `sendMessage`. `draftStreamer.ts`.
  - `edit`: legacy throttled `editMessageText` of a placeholder + typing indicator. `streamer.ts`.
  - The two draft backends share `baseDraftStreamer.ts` (throttled flush + 20s keepalive so the 30s ephemeral preview doesn't lapse; stable non-zero `draft_id` so updates animate). Drafts are **private-chat only** and **ephemeral**, nothing persists until the `finalize` send. `send.ts` holds the shared final-send (markdown→HTML, 4096 split, plain-text fallback on "can't parse entities"); `formatting.ts` does the markdown→HTML conversion used by the non-rich paths.
  - Bot API 9.3/10.1 methods have no telegraf wrapper (4.16.3), so they're called raw via `tg.callApi(<method>, …)` (typed loosely as `RawApi`).
- **Memory** (`src/core/memory.ts` + `src/mcp/memory.ts`): a durable, self-accumulating fact store (JSON `memory.json`). The `MemoryStore` singleton (`memory`) holds entries `{id, text, tags, salience, tier, useCount, createdAt, updatedAt, lastUsedAt, embedding?, embeddingModel?}` live in-process (like `WorkerManager`) and persists on mutation. Exposed to the agent as an in-process MCP server (`memoryMcp`, tools `mcp__memory__memory_{write,search,list}`, all in `AUTO_ALLOWED_TOOLS`) registered in every `runTurn` site (bot/chat/workers). `runTurn` itself calls `memory.recallForPromptAsync(prompt)` each turn and folds the top matches into the system prompt via `systemPrompt(append, memories)` (`prompt.ts`'s "# Relevant memories" section), so recall is automatic and store is frictionless. Recall always injects every `hot` entry plus the top relevance-matched `warm` entries; crucially `finishRecall` only bumps `useCount`/`lastUsedAt` for entries that were a genuine relevance hit for that prompt (`hitIds`), **not** for hot entries merely auto-injected — otherwise a hot entry's idle timer would refresh every turn and it could never decay. `prompt.ts` + the `memory_write` tool description steer the agent to keep each entry to one terse sentence (~150 chars) and to default to `warm` (records of work), reserving `hot` for a few always-relevant standing facts, since hot costs context on every turn. Panel **Memory** view (`panel/src/components/Memory.tsx`, REST `/api/memories` GET/POST/PUT/DELETE with `?q=` search) lists/searches/edits/deletes (the raw embedding vector is stripped from responses, replaced with an `embedded` boolean). **Semantic search (Phase 2)** lives in `src/core/embeddings.ts`: an off-by-default local embedding client (`EMBEDDING_*` config, `ollama` POST `/api/embeddings` or `openai`-shaped POST `/v1/embeddings` for LM Studio/OpenAI/proxies, with an optional `vault:` auth token). When `EMBEDDING_ENABLED=true`, `create`/`update` embed entries in the background (fire-and-forget), `index.ts` kicks off a startup `ensureEmbeddings()` backfill, and `semanticSearch()`/`recallForPromptAsync()` rank by a blend of cosine similarity (0.7) + normalized keyword overlap (0.3) + salience (0.1). Everything is best-effort: when embeddings are off, the endpoint is down, or no vectors exist yet, it falls back to the keyword `search()`. `embeddingModel` tags each vector with `provider:model` so a model change invalidates and recomputes stale vectors. Tier decay (hot→warm→cold) + compaction shipped (`maintenance.ts`): deterministic demote/delete by salience, then a Haiku `consolidateTier` pass over the hot and warm tiers (hot first, since it injects every turn) that finds entries stating the same fact in different words, rewrites each group into one consolidated entry via `memory.update` (which re-embeds it), and drops the redundant copies. A following `shortenVerbose` pass (hot then warm) condenses any single entry longer than `MEMORY_SHORTEN_CHARS` (default 220, `0` disables) into one terse sentence via Haiku that keeps every distinct fact, so a wordy lone entry with no duplicate twin still gets trimmed to keep recall context small (it only accepts a rewrite that is genuinely shorter and non-empty). Tracked in `MaintenanceStats` as `memoriesMerged` (dropped), `memoriesRewritten` (dedup rewrites), and `memoriesShortened` (verbose entries condensed). The AI step (`callHaiku`) runs through the Agent SDK `query()` with the haiku model, so it uses the bot's Claude connection (CLI login or `ANTHROPIC_API_KEY`) rather than a separate key; it returns null and only the deterministic steps run if that call fails. `runOnce()` is what the panel "Run now" (`POST /api/maintenance/run`) calls. A **dry-run preview** (`previewCompaction()`, `POST /api/maintenance/preview`) runs only the deterministic salience-threshold passes without mutating any entry, returning `MaintenancePreview` (`toDelete`, `toDemote`) so the panel Health card can show a collapsible preview of what would happen before the next run fires. Scheduling is interval-based and downtime-robust: the scheduler ticks every 60s and `mode()` resolves `MAINTENANCE_CRON` into one of three modes — **unset (default) → interval**: run whenever `now - lastRunAt >= 24h` (or never run), so a long-overdue run catches up on the first tick after boot (`start()` fires an immediate `checkAndRun()`); **`HH:MM` → cron**: fire daily at that server-local clock minute, guarded by a `MIN_RUN_GAP_MS` (~23h) so a clock match can't double-fire; **`off` → disabled**. Stats persist to `maintenance.json` (so last-run survives restarts) and `view()` adds a computed `nextRunAt` (interval: `lastRunAt + 24h`; cron: next `HH:MM`; `off`: undefined), both shown in the panel Health card.
- **Connectors** (`src/core/connectors.ts`): a placeholder catalog (Gmail, Google Calendar, Google Drive, Notion, Apple Calendar, Apple Mail, no Slack, Telegram is our channel) marked `coming-soon`. The registration surface + a vault-backed credential slot exist (`/api/connectors`, panel Connectors view lets you pre-attach a `vault:<id>` secret) but none are wired to a live MCP server yet.
- **Skill factory** (`src/mcp/skills.ts`): the agent can distil reusable procedures into the skills library itself via `skillsMcp` (`skill_save` upserts by name, `skill_patch` appends, `skill_list`; auto-allowed, registered in every run site). A prompt nudge (`prompt.ts`) tells it to save durable facts (`memory_write`) and reusable workflows (`skill_save`) sparingly. Saved skills show up in the existing panel Skills view. (The roadmap's alternative, a post-turn autonomous extraction pass, was deliberately skipped in favour of this deliberate-tool approach.)
- **Kanban delegation** (`src/core/taskRunner.ts` + `src/mcp/tasks.ts`): cards (`tasks.ts`, now with `priority`, `parentId`, per-column WIP limits, and a live `delegate` state) can be **delegated to an autonomous agent run**, `taskDelegator.delegate(id)` runs the card's title+notes via `runTurn` (`bypassPermissions`, memory+tasks MCP), moves it backlog→doing, streams output to panel clients (`{type:"task", …}` over the hub), and lands it in done (or records an error on `delegate`). The agent can also manage the board itself through the `tasksMcp` server (`task_create`/`list`/`update`, auto-allowed, registered in every run site), which is also how **auto-breakdown** works (a delegated run creates subtasks via `task_create` with `parentId`). Panel Tasks view adds WIP counts, priority, card-aging tint, and a Delegate/Stop control. Routes: `/api/tasks/wip`, `/api/tasks/:id/delegate|stop`.
- **Secret vault** (`src/core/vault.ts`): AES-256-GCM encrypted secrets in `vault.json`; the 32-byte master key lives in the macOS Keychain (`security` CLI, service `cct-vault`) with a `0600` `vault.key` file fallback on Linux. Secrets are referenced anywhere a token is stored as `vault:<id>` and resolved at use-time via `resolveSecret()` (provider auth tokens in `workers.ts`/`mainSettings.ts`/`status.ts` + the model-list routes). `importProviderSecrets()` ("Scan & import") moves plaintext provider tokens into the vault and rewrites them to references. Panel **Vault** view + `/api/vault` CRUD, `/api/vault/:id/reveal`, `/api/vault/import`. `list()`/`SecretView` never carry plaintext (only a masked `hint`). Config-level `.env` token migration is deliberately out of scope (would need the vault at config-parse time).
- **Heartbeat** (`src/core/heartbeat.ts`): proactive monitoring, **off by default**. The `heartbeat` singleton ticks every 60s but only evaluates every `config.intervalMs`; `bot.ts` injects `notify` (messages every `allowedUserIds`) and `runActive` (an autonomous turn to the first allowed user, skipped if busy). Modes: `off`; `alert` = deterministic host-health (CPU/mem/swap/disk thresholds via `getHealth`) + stalled-kanban-card checks, messaging on breach with a 3h per-signal cooldown; `active` = hands the detected signals to an autonomous agent turn so it can investigate/act and message only if noteworthy. Config + recent alerts persist to `heartbeat.json`. Panel **Heartbeat** view + `/api/heartbeat` GET/PUT + `/api/heartbeat/run`.
- **Status** (`src/core/status.ts`): `/api/status` returns the **public Claude service status** (Statuspage `status.claude.com/api/v2/status.json`, needs no credentials) plus per-backend probes for every configured provider and any running local backend (LM Studio :1234 / Ollama :11434, surfaced only when reachable). Anthropic itself isn't probed here, the public service banner already covers it. Panel **Status** view (in the **Others** nav group) polls it every 15s. Reuses `providerModels.ts` for model listing.
- **MCP send_file** (`src/mcp/sendFile.ts`): an in-process MCP server giving Claude a `send_file` tool to push files back to the chat. It's in `AUTO_ALLOWED_TOOLS` because it's a deliberate user-facing action.
- **Incoming files** (`src/telegram/files.ts`): uploaded docs/photos are downloaded into the session `cwd`, then a synthetic prompt tells Claude where the file landed. Images (`isViewableImage`) are additionally read back as base64 (`readImageInput`) and passed to the runner as inline vision content blocks, `runTurn` switches from a string prompt to a one-shot streaming-input `SDKUserMessage` (`imagePrompt` in `runner.ts`) when `images` are present, so the model sees the picture directly instead of via a `Read` round-trip.
- **Git review flow** (`src/git.ts` + `src/telegram/gitFlow.ts`): `/diff` renders the working-tree status+diff (sent as a `.diff` document when large) with inline Commit/Discard buttons; `/commit <msg>` stages and commits. Button presses are routed through the shared `callback_query` handler in `bot.ts` (`isGitCallback`/`resolveGitCallback`), namespaced `git:<action>`; Discard requires a second confirm press and only touches tracked files. All git calls go through `execFile` (no shell) and never throw, failures come back as `{ ok: false }`.
- **Usage tracking**: `runTurn`'s `result` cost/duration is folded into per-session lifetime + per-day buckets (`sessions.recordUsage`, persisted); `/usage` reports today + lifetime for the chat.
- **Approval presets** (`src/telegram/permissions.ts` + `canUseTool`): "Always allow" persists across restarts (it writes `session.sessionAllowedTools`, which is serialized). Bash approvals get an extra "always allow `<program>` commands" button whose leading token (`bashLeadCmd`) is stored in `session.allowedBashCmds`; `canUseTool` auto-allows a Bash call whose lead matches. `/allow`, `/allowed`, `/disallow` manage these. This is the middle tier between `safe` (prompt everything) and `auto` (prompt nothing).
- **Projects** (`src/telegram/projects.ts`): per-session list of saved cwds (`session.projects`, persisted). `/projects` posts an inline menu; `proj:` callbacks switch/add/remove and re-render the keyboard in place.
- **Voice** (`src/telegram/voice.ts` dispatcher): `TRANSCRIBE_PROVIDER` selects `openai` (download → POST to an OpenAI-compatible `/audio/transcriptions`, works with Groq too) or `vosk` (`src/telegram/vosk.ts`: ffmpeg decodes OGG/Opus → 16kHz mono PCM, fed to a lazily-loaded, cached Vosk model, `vosk` is an **optionalDependency**, imported via a non-literal specifier so the build doesn't require it). Either way the transcript is echoed and run as a normal text turn; `voiceEnabled()`/`voiceSetupHint()` gate it per provider.
- **Scheduling** (`src/schedule/`): `ScheduleManager` (singleton `schedules`) holds a persisted list (sibling `schedules.json`) and a 30s tick loop. `bot.ts` injects the runner via `schedules.start(...)`: a due job runs as an **autonomous** turn (`runUserPrompt(..., { autonomous: true, cwd })` → `bypassPermissions`) in the schedule's captured cwd, skipped (and retried next tick) if the chat is busy. `parseWhen` accepts `30m/2h/1d` (interval, ≥1m) or `HH:MM` (daily, server-local time). The per-turn `autonomous`/`cwd` overrides live on `handleUserPrompt`'s `TurnOptions`. Schedules are also managed from the panel Schedules view (`POST /api/schedules` with `prompt`/`when`/`cwd`, `DELETE /api/schedules/:id` via `schedules.removeById`); panel-created jobs default to the first `allowedUserIds` chat and `WORKDIR`.

- **Management panel** (`src/panel/server.ts` + `src/core/` + `panel/`): an optional embedded web dashboard, off unless `PANEL_ENABLED=true` (and it refuses to start without `PANEL_TOKEN`, enforced in `config.ts`'s `superRefine`). A Fastify server started in-process from `index.ts` (`startPanel`), so its handlers read the live `sessions`/`schedules` singletons directly, no IPC. Token auth gates everything under `/api`/`/ws`; static SPA assets are served freely. The token must be a `Bearer` header for REST; the `?token=` query string is accepted **only** for the `/ws` handshake (browsers can't set a WS header), so the secret never lands in proxy/access logs for REST calls (SEC-3). `config.ts`'s `superRefine` also enforces a 16-char minimum, and an in-memory per-IP lockout (`MAX_FAILS`=10 → 5min, cleared on success/restart, spoof-safe since Fastify runs with no `trustProxy`) throttles brute-force attempts. To stop that minimum from crash-looping an existing install whose token predates it, `healPanelToken()` (run inside `parseConfig` before validation, only when the panel is enabled) auto-heals a missing/short token: it generates a strong `base64url` token, rewrites the `PANEL_TOKEN` line in `.env` in place (mode `0600`), applies it to `process.env` for the live run, and records it in the exported `regeneratedPanelToken`; `index.ts` then DMs the new secret to every allowed user once the bot connects (the old one no longer works). Read-only dashboards: `/api/health` (a `systeminformation` snapshot, CPU/mem/swap/disk/IO, `src/core/health.ts`), `/api/sessions`, `/api/schedules`, `/api/usage` (views in `src/core/snapshot.ts`), `/api/audit`. Write features (all telegraf-free in `src/core/`, each persisting to its own JSON store in the data dir via `src/core/jsonStore.ts`, and appending to `audit.jsonl` via `src/core/audit.ts`; SEC-8 hardening: `ensureDataDir()` creates and `chmod`s the data dir to `0700` (it holds the vault key, session resume tokens, and memory at rest, so other local users mustn't read it) and is shared by `session/store.ts`'s state writer, and all store reads parse with a `safeReviver` that drops `__proto__`/`constructor`/`prototype` keys as defence-in-depth against a poisoned store file): **prompt** (`playbook.ts`, view the read-only personality + read/write `work.md`), **skills** (`skills.ts`, a reusable prompt library; `claudeFiles.ts`, a scoped browser/editor for on-disk `.claude/{agents,skills,commands}/*.md` + `CLAUDE.md` under known cwds, writes refused outside those paths; SEC-6 hardens the scope check by canonicalising every path with `realpathSync` (the deepest existing ancestor is realpath'd then the non-existent tail re-appended) so a symlinked component can't escape the root, and dropping too-shallow roots (`< 2` path segments, so a `/` session cwd can't widen the editable scope to the whole disk)), **tasks** (`tasks.ts`, a Trello board with `backlog/doing/done` columns + drag-drop reorder), and **workers** (`workers.ts`, persisted sub-agents). The `WorkerManager` (singleton `workers`) runs named autonomous agents via `runTurn` directly (own `AbortController`, `bypassPermissions`, optional per-worker `model` override and `systemPromptAppend` built from a skill + persona); runs are **concurrent** (one in-flight per worker, keyed in an `active` map, not gated by the per-chat `busy` flag), can be scheduled (own 30s tick reusing `parseWhen`/`nextRun`), and stream live output to all panel clients through the `PanelHub` (`src/panel/hub.ts`), which also owns the 2s health push on `/ws`. `runTurn` gained optional `model`, `systemPromptAppend`, and `env` options (`env` is merged over `process.env` for the spawned CLI; the SDK's `Options.env` supports this). **Providers** (`src/core/providers.ts`, `providers.json`) are named model-endpoint presets (base URL + auth token); a worker referencing a `providerId` runs with `ANTHROPIC_BASE_URL`/`ANTHROPIC_AUTH_TOKEN` set (and `ANTHROPIC_API_KEY` cleared) so it can target a **local model** (LM Studio, Ollama) or a proxy with a free-text model name, set globally instead via `.env` since the bot inherits `process.env`. The panel can list a provider's models server-side (`src/core/providerModels.ts`, OpenAI `/v1/models`, Ollama `/api/tags` fallback) via `POST /api/providers/models` (unsaved endpoint) / `GET /api/providers/:id/models` (saved), and offers LM Studio/Ollama prefill presets in the provider form. Every server-side outbound fetch built from a user-supplied URL (provider model listing + the embedding client in `embeddings.ts`) is run through `assertSafeUrl()` (`src/core/safeUrl.ts`, SEC-5) first: it enforces http(s), and blocks the cloud-metadata IP `169.254.169.254` / the rest of link-local (`169.254.0.0/16`, IPv6 `fe80::/10`) by resolving the host's DNS, while deliberately still allowing loopback + private LAN ranges (local models need them). A blocked URL throws `BlockedUrlError` (surfaced as a 400 by the model-list routes). Saved provider tokens are only ever attached to that same saved provider's own base URL, never to a caller-chosen URL, so a token can't be exfiltrated to an arbitrary endpoint. The panel API never returns a provider's plaintext `authToken` (SEC-2): `GET /api/providers` + the create/update responses go through `toProviderView()` (a `hasToken` boolean + masked `tokenHint`), and `updateProvider` treats a blank `authToken` as "keep existing" so an edit without re-typing the token leaves it intact. **Panel chat** (`src/core/chat.ts`, `panel/src/components/Chat.tsx`): a single persistent Claude session driven from the panel, its own resume token + cwd (`chat.json`), separate from any Telegram chat, streamed to all clients over the same `/ws` (frames `{type:"chat", event:…}`). It reuses `resolveMainRun()` so it tracks the main agent's model/provider. Default permission model is **safe**: `AUTO_ALLOWED_TOOLS` run automatically, anything else is held behind an in-panel **approve/deny** prompt (`canUseTool` broadcasts an `approval` frame and blocks until `POST /api/chat/approve` resolves it, or `APPROVAL_TIMEOUT_MS` denies). An **auto** (`bypassPermissions`) mode that skips prompts is only reachable when `PANEL_CHAT_BYPASS=true` in the env, otherwise the UI toggle is locked. The whole feature can be turned off with `PANEL_CHAT_ENABLED=false` (hides the view via `/api/me`'s `chatEnabled`, rejects `send`). REST: `GET /api/chat`, `/api/chat/send|stop|clear`, `PUT /api/chat/settings` (cwd + auto). The singleton is `chat`, wired via `chat.start(broadcast)` in `server.ts`.

The **main agent** (the bot driving chats) has the same runtime knobs: `src/core/mainSettings.ts` (`mainAgent.json`) persists an optional `model` + `providerId`, `resolveMainRun()` returns the per-turn `model`/`env` which `bot.ts`'s `handleUserPrompt` applies to every turn, so the whole bot can switch between Opus/Sonnet/a local model live (next message; each turn spawns a fresh CLI). Panel controls (`/api/agent` GET/PUT, `/api/agent/reset`, `/api/agent/restart`): reset aborts in-flight turns + clears every session's `sessionId` (`sessions.resetAll()`); restart shells `scripts/agentctl.sh restart` but only when `serviceInstalled()` is true (`src/core/agentControl.ts`), since otherwise the process would die without respawning. **`/model` command** (`src/commands.ts`): sends a 2-column shortcut grid (5 fixed Anthropic tiers, `claude-opus-4-8`, `claude-sonnet-4-6`, `claude-haiku-4-5-20251001`, plus two legacy) plus plain-text listing of any configured provider models (fetched server-side); `/model <name>` sets directly without the menu. Callbacks are namespaced `mdl:<modelid>`; noop header-row callbacks are handled by the `data.startsWith("mdl:noop")` branch. A **persistent logs** system (`src/logger.ts`): in addition to the in-memory ring buffer, every log entry is appended as NDJSON to `logs/YYYY-MM-DD.log` (one file per day, never truncated on restart); files older than 72h are deleted on startup and daily. Every entry is run through a central **secret redactor** (SEC-4) in `emit()` before it reaches any sink (console, ring, file, panel WS): `redactString`/`redactMeta` scrub `Bearer <token>`, `sk-`/`sk-ant-`/`pk-` key prefixes, Telegram `id:secret` bot tokens, `key=value`/`"key":"value"` pairs for sensitive keys (token, password, api_key, authorization, …), and `--token`/`--password` CLI flags. It's best-effort pattern matching (can't catch every secret) but it stops obvious credentials in tool-use command previews from being persisted; redaction never throws (meta object redaction falls back to the original value on any error). `availableLogDates()` lists existing log files; `readLogFile(date, {q, level, limit})` reads and filters one, returning the **whole file by default** (no cap) so a full day of logs is shown without silently dropping older lines, an explicit positive `limit` trims to the last N. The `/api/logs` endpoint accepts `?date=YYYY-MM-DD&q=&level=&limit=`, when `date` is present it reads the file (uncapped unless `limit` is passed); otherwise it returns the live in-memory ring buffer (capped at `RING_MAX`=1000). `/api/logs/dates` lists available dates. **Smart cross-file log analytics** (`logger.ts`): `searchAllLogs({q, level, sinceMs, limit})` walks every retained file (the whole 72h window) merging matches into one timeline (sorted oldest-first, optional `sinceMs` cutoff and `limit`); `summarizeUsage({sinceMs, top})` parses the persisted `"Tool use"` entries to tally most-used tools and (for Bash) the leading shell command token (`leadCommand` drops env-var prefixes and path components so `/usr/bin/node` and `node` fold together), returning a `LogUsageSummary`. Both share a `forEachLogEntry` iterator. Routes: `GET /api/logs/search?q=&level=&hours=72&limit=` (cross-file event search) and `GET /api/logs/summary?hours=72` (usage insights). The panel Logs view (`panel/src/components/Logs.tsx`) is organised into three tabs (so the analytics no longer squeeze the log box, the previous inline-Insights layout problem): **Activity** (`ActivityFeed`/`toActivities`/`describeTool`) a human-readable live feed that derives friendly rows — icon + verb ("Reading", "Running command", "Editing", …) + arg preview + timestamp — from the persisted `"Tool use"` entries (`meta.tool`/`meta.arg`), follows live like the raw view, and is the default tab; **Logs** (`RawLogs`) the original raw text view with the date dropdown (incl. the **All files (72h)** cross-file option), keyword search, and per-level toggles; and **Analytics** (`AnalyticsTab`) the full-height usage-insights panel (ranked most-used tools and shell commands, lazily loaded when the tab is first opened). A **live logs** view (`/api/logs` backlog + `{type:"log"}` WS frames) tails `logger.ts`'s in-memory ring buffer, `logger.ts` exports `recentLogs()`/`onLog()`, and the panel subscribes and rebroadcasts through the hub (there is no PTY to attach since the SDK spawns `claude` with piped stdio, so this ring buffer is the "remote view"). The frontend (`panel/`) is a separate React+Vite+Tailwind v4 app (semantic theme tokens in `index.css` give light/dark) built to `panel/dist` (served by Fastify in prod; `vite dev` proxies `/api`+`/ws` in dev). `npm run build` builds the panel first, then `tsc`. Bind defaults to loopback, expose remotely only behind a reverse proxy / private network. **Graceful shutdown** (`src/index.ts`): SIGTERM stops schedulers, waits up to 30s for in-flight turns (`whenSettled`), then holds 10s before flushing and exiting. Total worst-case: ~43s. `launchd ExitTimeOut=85`, `systemd TimeoutStopSec=85`.

The SDK is configured with `settingSources: ["user", "project", "local"]` so the driven agent loads real CLAUDE.md / settings from whatever `cwd` it runs in, i.e. this bot behaves like a genuine Claude Code session in the target project, not a sandbox.

- **System prompt / personality** (`src/prompt.ts`): each turn passes `systemPrompt: { type: "preset", preset: "claude_code", append }`, keeping Claude Code's defaults while appending a fixed personality plus the contents of `work.md` (path overridable via `WORK_FILE`). `work.md` is an operator playbook (how to handle recurring ops requests, services, crontab, deploys) re-read every turn, so edits apply without a restart.

`src/claude/events.ts` holds narrow type guards over the SDK's loosely-typed message union; prefer adding/using a guard there over inline `any` casts when reading new SDK message fields.

## Fleet build subsystems

The sections above describe the original single-bot core. The features below were layered on top of it. They follow the same conventions (telegraf-free logic in `src/core/`, an in-process MCP server per agent capability, a panel view plus REST routes per feature, JSON persistence in the data dir).

- **Crew hierarchy** (`src/core/workers.ts`, `src/telegram/leadBot.ts`, `src/mcp/crew.ts`): the `WorkerManager` singleton (`workers`) runs named autonomous agents (Leads, Assistants, specialists) via `runTurn`. A worker with a `role` of "lead" and a `telegramToken` (a `vault:<id>` reference) gets its own slim Telegraf instance (`leadBot.ts`) with its own session and the same auth: both the main bot and every Lead bot gate updates through the shared `isAuthorized(ctx)` helper (`src/auth.ts`, SEC-7), which requires an allow-listed sender **and** a private 1:1 chat whose id equals the sender's id, so a Lead bot added to a group can't leak host paths/command output to other members. Lead bots start and stop **live** when a worker is created, enabled, or disabled from the panel (no restart required): `workers.ts` calls `leadBotManager.start(worker)` / `.stop(id)` on mutation so the Telegraf polling loop begins or ends immediately. `prompt.ts` injects the live Lead roster into Atlas's system prompt each turn. The `crewMcp` server gives agents `crew_delegate` (hand a subtask to a Lead, get output back), `crew_report` (log a summary, optionally notify the president), `crew_ask_president` (pause the turn until the user replies, state held in `crewAsk.ts`), and `crew_suggest` (file a non-urgent proposal to the president's suggestion inbox, queued for triage — the president accepts it as a task or dismisses it from the panel Crew tab; acceptance creates a kanban card). The `crewMcp` is registered in **every** run site that talks to a human chat — the main bot (`bot.ts`) and each Lead bot (`leadBot.ts`, `notify`→its own chat, `primaryChatId`→`ctx.chat.id`). `crew_ask_president` registers a pending ask keyed by `primaryChatId`; the **plain text reply** that resolves it is matched in the chat's text handler via `hasPendingAsk`/`resolveAsk` **before** the busy guard (the asking turn holds `busy=true`, so the reply must short-circuit rather than start a new turn) — wired in both the main bot and the Lead bot. **`AskUserQuestion` inline buttons**: when an agent calls `AskUserQuestion`, `AskQuestionManager` (`src/telegram/askQuestion.ts`) renders the options as Telegram inline buttons (callback ns `ask:<id>:<idx>`) and a free-text fallback row; tapping a button resolves the question and deletes the keyboard. The manager is instantiated in both `bot.ts` and `leadBot.ts` over the respective `bot.telegram`. Lead bot final replies are split on `---`, mirroring the main bot's finish UX, so the work log collapses and only the closing sentence is shown as the chat bubble. Panel Crew and Workers views, plus `/api/workers*`, `/api/runs`, `/api/delegations`.
- **Council votes** (`src/core/council.ts`): `/council <proposal>` (or `POST /api/council` from the panel Crew tab) runs every enabled Lead as a one-shot vote (SUPPORT or OPPOSE with a reason and a concern), tallies the result to Telegram, and stores the session. Read back via `GET /api/council`.
- **Autonomy levels** (`src/core/mainSettings.ts`, per-worker config): the `safe`/`auto` split documented above is superseded by four levels, `supervised` (every tool prompts), `standard` (read-only tools auto-run, risky ones prompt; the default), `full` (`bypassPermissions`), and `auto_until_error`. Set globally per agent, per worker, or per chat with `/mode`. The mapping to SDK `permissionMode` lives alongside `canUseTool`. **`auto_until_error`** (`src/session/manager.ts` `AUTO_UNTIL_ERROR_TOOLS`/`AUTO_UNTIL_ERROR_COOLDOWN` + `noteToolError`/`resetEscalation`): auto-approves the read-only safe set *plus* a trusted write set (Bash/Write/Edit/NotebookEdit) until a tool returns an error, then opens a supervised cooldown (the next N=3 calls prompt the user) before resuming auto-approval. It runs with SDK `permissionMode: "default"` so `canUseTool` is consulted every call; the runner detects per-tool failures via a new `onToolResult(isError)` callback (`runner.ts` parses `user`/`tool_result` blocks for `is_error` through `hasToolError` in `events.ts`), and `bot.ts` calls `sessions.noteToolError(chatId)` to open the cooldown. The escalation state (`session.escalation.cooldown`) is transient (not persisted) and reset at each turn start. It is a per-chat interactive mode (main bot + panel), not offered to autonomous workers/leads.
- **Personas and language** (`src/core/mainSettings.ts`, `src/core/languages.ts`): each agent has an optional `persona` (character and tone, separate from the system prompt's domain knowledge) and `defaultLanguage` (BCP 47, from a 30-language catalogue). `/lang <code>` overrides per chat. Both fold into `prompt.ts`.
- **Plan and usage** (`src/core/planSettings.ts`, `src/core/usageProbe.ts`, `src/core/claudeUsage.ts`): subscription plan (Pro/Max/API), monthly cap, billing day, and alert threshold. `usageProbe.ts` reads live 5h session and 7d weekly limits from the Anthropic OAuth endpoints using the Keychain token the Claude CLI stores; `claudeUsage.ts` reads historical stats from `~/.claude/stats-cache.json`. Routes `GET|PUT /api/plan`, `POST /api/plan/report-test`, `GET /api/usage-probe`, `POST /api/usage-probe/run`, `GET /api/claude-usage`.
- **Semantic memory** (`src/core/embeddings.ts`): the local embedding client behind the `EMBEDDING_*` config. The default is auto mode (`_auto`, persisted in `embeddings.json`): `autoProbeEmbeddings()` runs at startup (`index.ts`), probes Ollama (`:11434`) then LM Studio (`:1234`) in preferred-backend order, and enables embeddings against whichever is live, re-selecting on every boot. It is skipped only on an explicit choice: the tri-state env `EMBEDDING_ENABLED` (`envEmbeddingMode()`) is `on` (pins the configured backend) or `off` (forces embeddings off and locks the panel), or auto mode has been left (`!_auto`). Leaving auto mode happens whenever a panel save, a panel disable, or a one-click connect calls `setEmbeddingsEnabled(..., auto=false)`. In auto mode, if no backend is reachable it persists embeddings off but stays auto, so a later boot re-enables when a backend returns. The panel exposes an Auto / Manual / Off control: Auto calls `POST /api/agent/embeddings/auto` (`enterAutoMode()`, which clears the pin and re-probes), Manual pins via `PUT /api/agent/embeddings`, and Off disables. When the env mode is `on`/`off` the control is locked and shows the env state. When enabled, `memory.ts` embeds entries in the background and `recallForPromptAsync()` ranks by a blend of cosine similarity, keyword overlap, and salience, falling back to keyword search when the endpoint is down. The agent card and Model Providers card in `Settings.tsx` probe Ollama (`/api/integrations/ollama`) and LM Studio (`/api/integrations/lmstudio`), offer one-click connect (`.../connect`), and pick a preferred backend (`PUT /api/agent/embeddings/preferred`) when both run. Embeddings config is `GET|PUT /api/agent/embeddings`.
- **Session resume after restart** (`src/telegram/resumePrompt.ts`): `SessionManager.isFirstSinceRestart(chatId)` is true once per chat per process when a resumable `sessionId` was rehydrated from disk. `maybeOfferResume` posts an inline resume-or-fresh offer and holds the prompt, auto-resuming after 10s. Autonomous turns call `markSeen` so they do not trigger the offer.
- **Task columns** (`src/core/columnConfig.ts`): the Kanban board's columns are user-defined (default `backlog`/`doing`/`done`), each with an optional WIP limit. Routes `GET|POST /api/tasks/columns`, `PUT|DELETE /api/tasks/columns/:id`, `POST /api/tasks/columns/reorder`, `POST /api/tasks/reorder`, `PUT /api/tasks/wip`.
- **Agentic loop detection** (`src/core/loopDetector.ts` + `src/telegram/loopPrompt.ts`): a per-turn `LoopDetector` (fresh instance per `handleUserPrompt`) hashes each `(tool name + input)` and counts repeats. When a call hits `LOOP_THRESHOLD` (config, default 3; `0` disables), the guard fires. For interactive turns it runs at the top of `canUseTool` (before the normal permission flow, so it catches even auto-allowed Read/Grep/Bash loops) and `LoopPromptManager` posts a Skip / Approve once / Continue prompt (callback ns `loop:<id>:<action>`, times out to "continue"): **Skip** denies that one call, **Approve once** pulls the counter back to `threshold-1` so the next repeat re-prompts, **Continue** silences that exact call for the rest of the turn. For `full`-autonomy turns the SDK runs `bypassPermissions` and never calls `canUseTool`, so detection instead happens in `onToolUse`, on a loop it notifies the chat and aborts the turn (`session.abort`), with `loopAborted` suppressing the generic "Stopped." message. Catches runaway retries burning tokens overnight; no new persistence or infra.
- **Auto skill extraction** (`src/core/autoSkill.ts`): gated by `AUTO_SKILL_GENERATION=true`, an async haiku pass after expensive turns proposes a skill entry. This is the alternative to the deliberate `skill_save` tool, not a replacement for it.
- **Panel terminal** (`src/panel/server.ts` + `src/core/ptyManager.ts`): a real shell session streamed over `/ws`. Routes `GET /api/terminal`, `POST /api/terminal/spawn`, `POST /api/terminal/resize`. **Off by default** (`PANEL_TERMINAL_ENABLED`, SEC-1, since a panel-token holder otherwise gets arbitrary host execution): when disabled `ptyManager.enabled` is false, `available` is false, the spawn/resize routes 403, and WS input is ignored, so the UI shows a "terminal disabled" hint (`reason: "disabled"` vs `"unsupported"` from `/api/terminal`). When enabled the shell gets a **minimal sanitized env** (`buildShellEnv()`: PATH/HOME/USER/SHELL/TERM/LANG allow-list) so a session can't read the bot's secrets back out via `env`; `PANEL_TERMINAL_INHERIT_ENV=true` restores the full process env (risky). A loud warning is logged at start when the terminal is enabled while `PANEL_HOST` is non-loopback.
- **In-panel updates** (`src/core/` update logic): check, apply, and roll back versions, mirroring `scripts/update.sh`. Routes `GET /api/update`, `POST /api/update/check|run|restore`.
- **Remote access / tunnel relay** (`src/core/tunnelManager.ts` + panel Remote Access view): exposes the loopback panel to the internet so it's reachable from a phone, still behind the panel login. The `tunnelManager` singleton spawns a relay binary (ngrok via `ngrok http <port>` with the authtoken passed through `NGROK_AUTHTOKEN` env so it never lands in argv, or cloudflared via `cloudflared tunnel --url`) as a child process pointed at `PANEL_PORT`, scrapes the public `https://` URL out of its stdout/stderr (per-provider `matchUrl`), and broadcasts state (`{type:"tunnel", view}`) to panel WS clients. A `ProviderSpec` abstraction (`command()`/`matchUrl()`) keeps it extensible. Config (provider/token/domain/`autoStart`/`basicAuth`/`passwordRef`) persists to `tunnel.json`; the token is a `vault:<id>` reference resolved at start-time and never returned to the client (`hasToken` boolean only). Same security posture as the terminal: **off by default** behind `PANEL_TUNNEL_ENABLED` (the public URL is host-equivalent), and even when enabled the relay only runs on an explicit Start from the view (or on **auto-start**, `autoStart`, default on, which relaunches it from `tunnelManager.start()` after a reboot/update — `start_relay()` no-ops gracefully when it can't launch). An **HTTP Basic Auth gate** (`basicAuth`, default on) sits in front of the public URL as a second factor independent of the panel token: the username is fixed (`BASIC_AUTH_USER` = `myhq`), the password is auto-generated (`randomBytes(12).toString("base64url")`) into the vault on first enable (`ensurePassword()`, so the tunnel is never exposed without a credential to check), revealable/rotatable from the view (`revealPassword`/`rotatePassword`/`setPassword`, min 6 chars), and constant-time compared (`verifyBasic`/`safeEqual` via `timingSafeEqual`). The gate is enforced in a Fastify `onRequest` hook that only applies to **tunnel traffic** (detected by the relay's `x-forwarded-for`/`x-forwarded-host` headers, so direct loopback/LAN panel access is never blocked) and `tunnelManager.basicAuthActive`, returning 401 + `WWW-Authenticate: Basic`. The freshly auto-generated password is DM'd to every allowed user over Telegram (`tunnelManager.setNotifier()` wired in `index.ts` before `startPanel()`). The Telegram `/status` command also surfaces the provider + public URL + login when the relay is running. Routes `GET|PUT /api/tunnel`, `POST /api/tunnel/start|stop`, `GET /api/tunnel/password`, `POST /api/tunnel/password` (start/stop/PUT/password 403 when disabled). Relay binaries are not bundled (ENOENT surfaces an "install the CLI" hint).
- **Branding** (`ATLAS_NAME`, `BRAND_NAME`): rename the main agent and the product for self-hosted deployments.

The panel REST surface is catalogued (with `curl` examples) in `work.md` under "Fleet API (Panel)", and grouped in the README's "Panel API" section. Keep all three in sync when adding or renaming a route.
